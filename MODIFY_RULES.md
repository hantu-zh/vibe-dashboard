# MODIFY_RULES.md - 文件修改控制规则

**目的**：防止意外破坏用户已确认的稳定版功能

---

## 🚨 核心规则：稳定版文件修改前必须询问

### 什么是"稳定版文件"？

任何包含以下标记的文件：
```html
<!-- 
  STABLE VERSION - USER APPROVED [日期]
  DO NOT MODIFY WITHOUT USER CONFIRMATION
  ...
-->
```

或文件名包含 `_stable_` 的备份文件。

---

## 修改稳定版文件的强制流程

### 1️⃣ 检查稳定版标记
- 修改任何文件前，**必须先读取文件前50行**
- 检查是否有 `STABLE VERSION` 标记
- **如果有标记 → 停止，进入询问流程**

### 2️⃣ 询问用户
**标准询问格式**：
```
⚠️ 文件修改确认
文件：news.html
状态：稳定版（2026-06-25 用户确认）
原因：[说明为什么要修改]

❓ 确定要修改吗？
- 输入"确定" → 继续修改
- 输入"取消" → 放弃修改
```

### 3️⃣ 用户同意后
- **移除文件头部的 `STABLE VERSION` 标记**（或改为 `UNDER MODIFICATION`）
- 进行修改
- 修改完成后 **立即测试**

### 4️⃣ 修改完成，等待用户验证
**标准回复格式**：
```
✅ 修改完成，等待验证

文件：news.html
修改内容：[简要说明]
状态：等待用户测试

请测试后告诉我结果：
- 如果正常 → 我重新添加稳定版标记
- 如果有问题 → 立即从 backups\stable\ 恢复
```

### 5️⃣ 用户验证通过
- 添加 `STABLE VERSION` 标记（更新日期）
- 创建新的备份快照
- 提交到 git（标签：`STABLE-[文件名]-[日期]`）

### 6️⃣ 用户验证失败
- **立即恢复**备份：`copy backups\stable\news_stable_2026-06-25.html news.html`
- 告知用户已恢复
- 分析问题，提出新方案，**再次询问**

## 例外情况（无需询问）

以下情况可以直接修改，但**修改后必须告知用户**：

1. **纯注释/文档更新**（不改变功能）
2. **修复明显错误**（如拼写错误、死链）
3. **用户明确要求修改**（已经有上下文）

## 备份策略

### 自动备份
- 标记稳定版时 → 自动创建 `backups\stable\[文件名]_stable_[日期].bak`
- 每天自动备份 → `backups\daily\[文件名]_[日期].bak`（heartbeat 任务）

### 手动备份
用户说"创建备份"时 → 立即执行备份并确认

---

## 违规处理

如果我（AI）违反此规则：
1. 用户说"回滚" → 立即从 `backups\stable\` 恢复最新稳定版
2. 分析违规原因 → 更新此文档
3. 向用户道歉 😅
